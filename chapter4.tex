\section{数据预处理与知识库构建}
课程助教系统的知识来源主要是 C 语言教材及相应的结构化文档，系统首先要将原始课程材料加工成可检索可引用可组织的知识库。在数据预处理阶段，系统对Markdown、PDF、PPT等格式的教材进行统一的结构化解析。程序通过正则表达式识别标题，并维护一个 heading_stack 来追踪当前文本所处的章节层级，当读取到正文块时，将其与当前的章节路径绑定，生成包含正文内容、来源文件、章节标题和完整章节路径的结构化文档，为后续回答提供精确到章节的引用依据，使学生能够沿着系统给出的来源信息直接回到教材原文中核验。

系统通过短文本合并机制来抑制切分过碎的问题，通过 merge_short_markdown_docs() 函数检查相邻文本块是否隶属于同一文件、同一条章节路径，结合 MIN_DOC_CHARS 和 MAX_DOC_CHARS 控制合并的粒度，将过短且合并后不超过上限的块拼合为完整的文档单元，避免因单句成块导致语义断裂，防止后续检索召回的只是只言片语而缺失必要的上下文支撑。

完成结构化之后，系统进入文本切块阶段，程序对文档内容做 token 编码，按设定的 chunk_token_size 和  chunk_overlap_token_size 将长文档拆解为若干文本块。每个块均保留文档编号、块序号、文本内容等基础字段，并继承前一阶段生成的元数据，为保证标识稳定，系统依据这些内容计算哈希值，生成以 chunk- 为前缀的唯一标识，避免重复插入同一内容，为后续检索结果与教材原文之间建立了精确可回溯的映射关系。

知识库构建的过程中，每一条文档在入库前先经标准化处理，确保 content 字段非空且元数据完整可追溯，随后依据内容与元数据生成文档级哈希编号，系统在正式索引前会检查该编号是否已存在，将重复数据挡在门外，抑制索引冗余，为知识库的增量更新提供了稳定基础。索引组织采取文本库、向量库与图谱库并行铺开的方式，完整文档存入 full_docs，切分后的文本块归入 text_chunks，两者均依托 JSON KV 存储管理。向量库部分由 NanoVectorDBStorage 承担，文本经 embedding 模型批处理转换为向量，写入之前会校验当前向量的维度与已有索引是否一致，防止因模型切换导致索引失效。整个流程将文档消重、多模态索引和维度校验串联在一起，能稳定复用已处理内容并在后续检索中以多种形式提供支撑。

仅靠文本块向量索引难以捕捉课程知识点之间显式的关联结构，AITA系统在此基础上进一步构建了实体关系图谱。在层次化模式下，HiRAG的插入流程会调用extract_hierarchical_entities()对文本块进行更细粒度的实体与关系抽取；若不启用层次化，则回退至普通实体抽取函数。抽取出的实体与关系既写入图存储，也进入实体向量库——前者承载知识点之间的连接关系，后者则用于后续的局部语义召回。图存储由NetworkXStorage实现，以节点表示实体或知识点，以边表示关系，最终落地为graph_chunk_entity_relation.graphml文件。

图谱建成后，系统引入Leiden聚类算法对实体网络进行社区发现，将联系紧密的节点归并为若干知识社区。随后，generate_community_report()为每个社区生成摘要信息并写入社区报告存储。这一步的意图，是为全局检索提供更高层次的知识概括：当学生提出的问题需要跨越若干章节或串联多个概念时，系统能够从社区报告中汲取全局语义线索，而不仅仅返回零散的文本片段。

最终，系统在workdirs下生成一套内容互补的知识库产物：kv_store_full_docs.json保存原始结构化文档，kv_store_text_chunks.json支持片段级引用，vdb_chunks.json和vdb_entities.json分别支撑文本与实体的语义检索，graph_chunk_entity_relation.graphml维护实体关系网络，kv_store_community_reports.json则提供知识社区的摘要视图。全文、片段、向量、关系与社区报告五者相互配合，将静态教材材料转化为可被检索、推理与引用的课程知识基底，为后续的助教问答提供了多粒度的数据支撑。
