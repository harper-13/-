\chapter{相关技术介绍}

\section{大语言模型技术}

LLM 代表了自然语言处理领域一条以规模扩展换取能力涌现的技术路径，其核心范式是在海量的文本语料上进行自监督预训练，模型从中学习自然语言的统计规律与深层语义结构来完成理解与生成任务。当参数规模突破某个量级之后，这些模型在文本生成、知识问答、推理分析乃至多轮对话交互中开始展现出较强的泛化能力，逐步成为当代人工智能技术栈中的基础性组件。这一能力跃升的背后，是序列建模架构的根本性变革。早期语言模型所依赖的循环神经网络（RNN）及长短期记忆网络（LSTM），在捕捉长距离依赖时受限于序列化计算与时序遗忘，难以胜任复杂语义的建模。Transformer架构通过自注意力机制重构了这一范式————它直接对序列中任意位置间的关联进行建模，打破了距离的壁垒，在大规模数据与参数条件下高效学习。由此催生的预训练模型家族，从GPT、BERT到LLaMA等，迅速将 LLM 推上了能力扩展的快车道。

预训练阶段，LLM 通过自回归或自编码两种路径来学习语言的分布规律，前者以前序词元预测下一个词元来驱动模型习得流畅的生成能力；后者则通过随机遮蔽并还原被掩码的词汇使模型建立更深层的语义理解。在获得语言基底之后，指令微调与对齐训练等技术引导模型将已有能力向理解用户意图与贴合任务要求的方向收敛，从而在特定应用场景中获得更稳定更可控的表现。

在教育场景中， LLM 能够以自然语言方式理解学生提出的问题，在对话中生成解释性的回答，缓解教师的答疑压力。但是课程内容具有明确范围和教学口径，仅依赖通用大语言模型难以保证与课堂讲授和教材表述保持严格一致，答案的出处也是不可被回溯核验的。为了提升可靠性，通常需要结合检索增强生成技术将模型与课程知识库结合，使模型能够在回答问题时引用真实资料并减少幻觉。

\section{检索增强生成技术}

RAG 技术在模型输出前插入了一个检索环节，使原本封闭的参数空间得以与外部知识库建立实时连接，回答不再只由模型内部的权重决定，而是基于可查阅可追溯的真实语料之上。这一特性使其天然适用于教育、医疗和法律等对于内容准确率要求高的领域。

从工作流来看，RAG检索的起点是查询构建，系统需要将用户的输入转译为适合检索的查询，并将其编码为向量表示，解决“用户想问什么”与“知识库里有什么”之间的语义对齐。随后是文档检索，系统在知识库中召回与查询最相关的片段，从海量材料中快速锁定可能包含答案的证据范围。在最终的答案生成阶段，大语言模型不再凭空作答，而是以检索到的文档片段为素材来组织回答。这一先检索、后组织的流程，将引用的依据从模型内部不可见的记忆转移为外部可以直接出示的语料来源，强制 LLM 在预设的课程内容或领域知识边界内说话，回答的可解释性得到结构性保障，它把知识更新与模型参数解耦，当课程内容或领域规范发生变化时，只需要更新知识库就可以让系统同步感知，不必重新经历成本高昂的模型重训练。对于需要与教学内容保持严格一致的教育场景而言，可维护性也是一种核心优势。

根据前述分析，RAG与教育场景的适配性及其边界已清晰可见，检索质量的上限直接决定了生成内容的可信度，一旦检索未能命中课程资料，模型便会退回依赖参数内部记忆的老路，幻觉依然可能重现；同时，知识库的清洗、分段与索引优化也构成持续的系统性投入。然而，教育的特性恰恰为这一机制提供了天然锚点——学生所提疑问并非随机发散，而是扎根于教材章节、实验步骤与课堂内容的具体语境，这为知识库的构建划定了清晰的取材边界，使RAG能够将回答约束在可追溯的课程原文之上，从而提升准确性与教学一致性。但我们也需清醒地看到，传统的RAG主要依赖向量检索来召回文本片段，这种扁平化的匹配方式在处理孤立事实性问题时足够有效，一旦面对需要跨章节概念关联、步骤间依赖推理或知识体系层级结构的复杂提问，片段式的检索便难以还原知识之间的内在脉络。正是为了突破这一限制，将知识图谱引入检索增强框架的Graph RAG技术进入了我们的视野。2.3节将重点探讨如何利用知识图谱对课程概念、实验操作及其前置关系进行结构化建模，使检索不再是孤立的语义匹配，而是沿着知识的内在骨架展开，从而在更复杂的答疑推理中提供更稳健的支撑。

上述分析表明 RAG 检索质量的上限直接决定了生成内容的可信度，一旦检索未能命中课程资料，模型便会退回依赖参数记忆的老路，幻觉仍可能重现，同时，知识库的清洗、分段与索引优化也需要持续投入成本。传统 RAG 依赖的向量检索本质上是一种扁平化匹配，面对需要跨章节概念关联、步骤间依赖推理或知识体系层级结构的复杂提问，孤立的文本片段难以还原知识之间的内在脉络。若要让系统在更深层的答疑推理中保持稳健，就不能止步于语义层面的片段召回，需要在检索中引入知识本身的结构。2.3节将重点讨论将知识图谱融入检索增强框架的 Graph RAG 技术如何对课程概念、实验操作及其前置关系进行结构化建模，使检索不再停留于孤立的语义匹配，而是沿着知识的内在骨架展开。

\section{ GraphRAG 技术 ？？待定}

GraphRAG 是一种将图结构知识与检索增强生成方法结合的技术框架。该方法通过构建知识图谱或图结构索引，使系统能够在回答问题时利用实体关系、主题结构和跨文档关联信息。GraphRAG 的目标是提升模型在复杂问题、多跳推理和结构化知识检索中的表现。

GraphRAG 的核心思想是将文本内容转化为图结构。系统会从文档中抽取实体、概念和关系，并将它们组织成节点和边。这样，知识库不再是独立的文本片段，而是一个具有结构化语义的图。模型在检索时可以沿着图结构找到相关节点，从而获得更丰富的上下文信息。

在技术流程上，GraphRAG 通常包括三个主要步骤。第一步是图构建。系统会对文档进行实体识别、关系抽取和主题聚类，并将结果组织成图结构。第二步是图检索。系统根据用户问题定位相关节点，并沿着图的连接关系扩展检索范围。第三步是答案生成。大语言模型在生成回答时会结合图检索得到的内容，使回答更准确、更具结构性。

GraphRAG 的优势主要体现在三个方面。第一，图结构能够表示跨段落、跨文档的关联信息，使系统更容易处理多跳推理任务。第二，图检索能够提供更清晰的知识路径，使回答更具可解释性。第三，图结构能够帮助系统识别主题关系，从而提升检索的覆盖范围和准确性。

但是，GraphRAG 也面临一些挑战。图构建过程依赖实体识别和关系抽取的质量，如果抽取结果不准确，图结构会受到影响。此外，图规模较大时，检索效率可能下降，需要额外的索引优化和图压缩策略。图的更新成本也较高，课程资料的变化会导致图结构需要重新构建。

在教育场景中，GraphRAG 能够帮助系统处理跨章节、跨知识点的问题。学生在学习过程中常常需要理解概念之间的关系，而图结构能够清晰表示这些关系。通过结合课程知识图谱，GraphRAG 可以使系统在回答问题时引用结构化知识，从而提升回答的准确性和解释性。这种方法能够弥补传统 RAG 在多跳推理和知识组织方面的不足，为构建更可靠的智能助教系统提供技术支持。

\section{知识图谱？？？}

知识图谱是一种以结构化方式表示实体及其关系的技术框架。该方法通过将概念、实体和属性组织成节点，并以关系连接这些节点，从而形成具有语义结构的图。知识图谱能够以清晰的方式表达知识之间的关联，使系统能够在检索、推理和问答任务中获得更丰富的语义信息。

知识图谱的核心思想是将非结构化文本转化为结构化知识。系统通常需要对文档进行实体识别、关系抽取和属性标注，并将这些信息构建为图结构。这样，知识库不再是孤立的文本片段，而是一个具有语义关联的网络。通过这种方式，系统能够在处理复杂问题时利用实体之间的关系，从而提升知识组织和检索能力。

在技术流程上，知识图谱构建一般包括三个主要步骤。第一步是知识抽取。系统从文本中识别实体、概念和关系，并对其进行分类和规范化。第二步是图构建。系统将抽取到的实体和关系组织成节点和边，并形成初步的图结构。第三步是图维护。系统需要对图进行更新、融合和去重，以保证图结构的完整性和一致性。

知识图谱具有三个主要优势。第一，图结构能够清晰表示知识之间的关联，使系统能够支持多跳推理和跨文档检索。第二，知识图谱能够提供可解释的知识路径，使系统回答更透明、更可追溯。第三，知识图谱能够支持结构化查询，使系统能够在复杂任务中获得更高的检索精度。

但是，知识图谱也存在一些挑战。知识抽取的质量会直接影响图结构的准确性。如果实体识别或关系抽取出现错误，图结构会受到影响。此外，图规模不断扩大时，图维护和检索的成本也会增加，需要采用图压缩、索引优化和增量更新等策略。知识图谱的构建和维护还需要持续投入，课程资料的变化也会带来更新压力。

在教育场景中，知识图谱能够帮助系统组织课程内容，并表示知识点之间的关系。学生在学习过程中常常需要理解概念之间的联系，而知识图谱能够以结构化方式呈现这些关系。通过将课程内容构建为知识图谱，系统能够在回答问题时引用结构化知识，从而提升回答的准确性和解释性。知识图谱还能够支持跨章节、跨主题的知识导航，为智能助教系统提供更强的知识组织能力。

总体来看，知识图谱技术为构建结构化知识库提供了重要基础。通过与检索增强生成技术和图检索方法结合，知识图谱能够进一步提升智能助教系统在复杂问题、多跳推理和知识组织方面的能力，为教育场景中的智能化支持提供关键技术保障。

\section{前后端开发技术？？？}

在智能助教系统的实现过程中，前后端技术为系统的交互体验、数据处理和服务响应提供了基础支持。前端主要负责界面展示和用户交互，后端负责业务逻辑、数据管理和模型调用。两者共同构成系统的整体运行框架，使用户能够以稳定、流畅的方式访问系统功能。

前端技术通常基于现代 Web 框架构建。常见的做法是使用组件化的前端框架来实现界面渲染、状态管理和交互逻辑。这样可以让系统在不同设备上保持一致的使用体验。前端还需要处理输入内容、展示模型回答、管理对话历史，并与后端保持实时通信。为了提升交互效率，前端一般会采用异步请求方式，使界面能够在不刷新页面的情况下更新内容。

后端技术承担系统的核心功能，包括模型调用、检索服务、知识库管理和用户请求处理。后端需要负责接收前端请求，完成问题解析、检索触发、模型推理和结果整合等任务。由于智能助教系统通常需要处理大量并发请求，后端框架需要具备良好的扩展性和稳定性。为了保证系统响应速度，后端还会对检索索引、缓存策略和数据结构进行优化，使模型能够在较短时间内返回结果。

在模型服务方面，后端需要提供统一的接口来管理大语言模型、RAG 模块和知识图谱模块。这样可以让系统在不同任务之间保持一致的调用方式。后端还需要处理模型输入格式、检索结果融合和输出内容过滤等问题，使系统能够在复杂场景中保持稳定表现。

前后端之间的通信通常通过 REST API 或 WebSocket 实现。REST API 适合处理普通请求，而 WebSocket 更适合流式输出和实时交互。对于智能助教系统来说，流式输出能够让用户更快看到模型生成的内容，从而提升使用体验。

在教育场景中，前后端技术不仅需要保证系统的稳定性，还需要支持课程资源管理、用户权限控制和数据安全等功能。前端需要提供清晰的界面来展示课程资料、实验指导书和历史对话记录。后端需要保证数据访问的安全性，并对不同角色（如学生、教师、助教）提供不同的权限设置。

总体来看，前后端技术为智能助教系统提供了基础运行环境。前端负责交互体验，后端负责核心逻辑，两者共同支持大语言模型、RAG 和知识图谱等模块的运行，使系统能够在教学场景中稳定、高效地提供服务。

\section{数据库与知识库技术？？？}

在智能助教系统中，数据库与知识库技术为数据存储、检索和管理提供了基础支持。数据库主要负责结构化数据的存储与访问，而知识库负责组织课程资料、文本内容和语义信息，使系统能够在检索和生成过程中获得可靠的知识来源。两者共同构成系统的底层数据基础，对系统的稳定性和回答质量具有重要影响。

数据库通常用于存储用户信息、对话记录、课程资源索引和系统日志等结构化数据。关系型数据库在数据一致性和事务管理方面具有优势，适合管理用户权限、课程结构和系统配置等内容。非关系型数据库在处理大规模文本、缓存数据和高并发访问时更具灵活性，因此常用于存储对话历史、检索缓存和模型中间结果。通过合理的数据库设计，系统能够在高负载情况下保持稳定响应。

知识库则承担更直接的知识支持功能。课程资料、教材内容、实验指导书和教学文档通常需要经过清洗、分段和索引，才能被系统有效使用。知识库不仅需要存储这些内容，还需要支持向量检索、关键词检索和图结构检索等多种方式，使系统能够根据不同任务选择合适的检索策略。为了提升检索效果，知识库还需要对文本进行分段、去重和格式规范化，使内容更适合模型处理。

在智能助教系统中，知识库的质量会直接影响回答的准确性。课程资料往往包含大量细节信息，而学生问题常常依赖具体段落、实验步骤或代码示例。因此，知识库需要保证内容的完整性和可追溯性，使系统能够在回答问题时引用真实资料。为了减少幻觉，知识库还需要提供明确的来源信息，使模型能够在生成回答时保持与课程内容一致。

数据库与知识库之间需要保持紧密协作。数据库负责管理知识库的索引、版本和更新记录，使系统能够在课程内容变化时及时更新知识库。知识库则为模型提供语义内容，使系统能够在回答问题时获得可靠依据。通过这种方式，系统能够在结构化数据管理和语义知识检索之间形成稳定的协同关系。

在教育场景中，数据库与知识库技术不仅支持日常答疑，还支持课程资源管理、学习行为分析和教学内容更新。数据库能够记录学生的提问行为和学习轨迹，为教学分析提供数据基础。知识库能够组织课程知识点，为智能助教系统提供可解释、可追溯的知识来源。两者共同提升系统的可靠性和可用性，为大规模教学环境中的智能化支持提供关键技术保障。

\section{系统测试相关技术？？？}

系统测试技术在智能助教系统的开发过程中具有重要作用。测试不仅用于验证系统功能是否正常，还用于评估系统的稳定性、性能和可靠性。由于智能助教系统涉及前端交互、后端服务、检索模块和大语言模型等多个部分，测试技术需要覆盖不同层面，才能保证系统在实际教学场景中稳定运行。

在功能测试方面，系统需要验证各模块是否按照预期工作。前端界面需要保证输入、展示和交互逻辑正常，后端服务需要保证请求处理、模型调用和检索流程正确。由于智能助教系统依赖知识库和模型推理，测试还需要关注检索结果是否准确、模型输出是否符合任务要求，以及系统是否能够正确处理异常输入。

性能测试主要关注系统在高并发场景下的响应速度和稳定性。智能助教系统在教学高峰期可能会同时接收大量学生提问，因此后端服务需要具备足够的处理能力。性能测试通常会模拟大量请求，观察系统的响应时间、吞吐量和资源占用情况。通过性能测试，可以发现系统在检索效率、缓存策略和模型推理速度方面的瓶颈，并为后续优化提供依据。

稳定性测试关注系统在长时间运行中的表现。智能助教系统需要在整个学期持续提供服务，因此系统必须能够在长时间运行后保持稳定。稳定性测试通常会模拟持续请求，观察系统是否出现内存泄漏、服务中断或性能下降等问题。通过稳定性测试，可以确保系统在真实教学环境中保持可靠运行。

在智能助教系统中，模型输出质量也是测试的重要内容。由于大语言模型可能出现幻觉或偏差，测试需要评估模型回答的准确性、相关性和一致性。常见做法是构建测试集，包含典型问题、边界问题和课程关键知识点，并对模型输出进行人工或自动评估。通过这种方式，可以发现模型在特定课程内容上的不足，并为后续优化提供参考。

此外，系统还需要进行安全测试。由于系统涉及用户数据、课程资料和权限管理，测试需要验证数据访问是否安全、权限控制是否正确，以及系统是否能够抵御常见攻击。安全测试能够保证系统在开放环境中运行时不会出现数据泄露或非法访问。

在教育场景中，系统测试不仅关注技术层面的正确性，还关注教学效果。测试需要验证系统是否能够准确理解学生问题、是否能够引用正确的课程资料、是否能够保持回答一致性。通过结合技术测试和教学测试，系统能够在功能、性能和教学质量方面达到更稳定的水平。

总体来看，系统测试技术为智能助教系统的可靠运行提供了重要保障。通过功能测试、性能测试、稳定性测试、模型质量测试和安全测试，系统能够在复杂教学场景中保持稳定、高效和可控的表现，为大规模教学环境提供可靠支持。
