\chapter{相关技术介绍}

\section{大语言模型技术}

LLM 代表了自然语言处理领域一条以规模扩展换取能力涌现的技术路径，其核心范式是在海量的文本语料上进行自监督预训练，模型从中学习自然语言的统计规律与深层语义结构来完成理解与生成任务。当模型参数规模和训练数据规模突破某个量级之后，这些模型在文本生成、知识问答、推理分析乃至多轮对话交互中开始展现出较强的泛化能力，逐步成为当代人工智能技术栈中的基础性组件\cite{touvron2023llama}。这一能力跃升的背后，是序列建模架构的根本性变革。早期语言模型所依赖的循环神经网络（RNN）及长短期记忆网络（LSTM），在捕捉长距离依赖时受限于序列化计算与时序遗忘，难以胜任复杂语义的建模\cite{elman1990finding}\cite{graves2012long}。Transformer架构通过自注意力机制重构了这一范式，直接对序列中任意位置间的关联进行建模\cite{vaswani2017attention}，打破了距离的壁垒，在大规模数据与参数条件下高效学习。由此催生的预训练模型家族，从GPT、BERT到LLaMA等，迅速将 LLM 推上了能力扩展的快车道\cite{brown2020language}\cite{devlin2019bert}\cite{touvron2023llama}。

在预训练阶段，LLM 通过自回归或自编码两种路径来学习语言的分布规律，前者以前序词元预测下一个词元来驱动模型习得流畅的生成能力，后者则通过随机遮蔽并还原被掩码的词汇使模型建立更深层的语义理解\cite{brown2020language}\cite{devlin2019bert}。在获得语言基底之后，指令微调与对齐训练等技术引导模型将已有能力向理解用户意图与贴合任务要求的方向收敛，从而在特定应用场景中获得更稳定更可控的表现\cite{ouyang2022training}。

在教育场景中 LLM 能够以自然语言方式理解学生提出的问题，在对话中生成解释性的回答，缓解教师的答疑压力。但是课程内容具有明确范围和教学口径，仅依赖通用大语言模型难以保证与课堂讲授和教材表述保持严格一致，模型输出的事实依据也难以直接追溯核验\cite{kasneci2023chatgpt}\cite{holmes2023guidance}。为了提升可靠性，通常需要结合 RAG 技术将模型与课程知识库结合，使模型能够在回答问题时引用真实资料并减少幻觉\cite{lewis2020retrieval}。

\section{检索增强生成技术}

RAG 技术在模型输出前插入了一个检索环节，使原本封闭的参数空间得以与外部知识库建立实时连接，得到的答案基于可查阅可追溯的真实语料之上\cite{lewis2020retrieval}。这一特性使其天然适用于教育、医疗和法律等对于内容准确率要求高的领域\cite{nguyen2022transformer}。

从工作流来看，RAG 检索的起点是查询构建，系统需要将用户的输入转译为适合检索的查询，并将其编码为向量表示，解决“用户想问什么”与“知识库里有什么”之间的语义对齐。随后是文档检索，系统在知识库中召回与查询最相关的片段，从海量材料中快速锁定可能包含答案的证据范围\cite{ru2024ragchecker}。在最终的答案生成阶段，LLM 以检索到的文档片段为素材来组织回答。这一先检索、后组织的流程，将引用的依据从模型内部不可见的记忆转移为外部可以直接出示的语料来源，强制 LLM 在预设的课程内容或领域知识边界内说话，回答的可解释性得到结构性保障，它把知识更新与模型参数解耦，当课程内容或领域规范发生变化时，只需要更新知识库就可以让系统同步感知，不必重新经历成本高昂的模型重训练\cite{pan2023large}。

上述分析表明 RAG 检索质量的上限直接决定了生成内容的可信度，一旦检索未能命中课程资料，模型便会回退至依赖参数记忆的老路，幻觉仍可能重现\cite{he2024g}。传统 RAG 依赖的向量检索本质上是一种扁平化匹配，面对需要跨章节概念关联、步骤间依赖推理或知识体系层级结构的复杂提问，孤立的文本片段难以还原知识之间的内在脉络，若要让系统在更深层的答疑推理中保持稳定，就需要在检索中引入知识本身的结构。

面对传统 RAG 在处理多跳推理与跨文档关联时的捉襟见肘，图检索增强生成（GraphRAG）将检索的底层结构从片段集合重构为关系网络，系统从相关文档中抽取实体、概念及其语义关系，并据此构建起以节点和边为骨架的知识图谱或图结构索引\cite{edge2024local}，检索过程从 RAG 里语义向量空间中点对点的相似度匹配转变成沿着实体间的定义、因果、前后置等显式关系边进行导航式跳转。当学生提出一个需要串联多个概念或跨越若干章节才能解答的复杂问题时，GraphRAG 能够循着图的拓扑结构，将分散在多处的证据收拢为一组有内在逻辑链条的上下文\cite{he2024g}。

GraphRAG 的技术实现可以拆解成三个彼此咬合的环节，在图构建阶段，系统对文档进行实体识别、关系抽取与主题聚类，将散落在各处的知识颗粒编织成为由节点和边构成的关联网络，在图检索阶段，系统定位到问题所触及的核心节点后沿图的连接关系进行导航式扩展，获取一组存在显性关联的上下文子图，天然支撑需要串联多个概念的多跳推理，至答案生成阶段，LLM 以图检索返回的结构化上下文为素材来组织答案，使得知识路径清晰可循而具备更高的可解释性。图结构对跨段落、跨文档关联的形式化表达从机制上拓宽了检索的覆盖边界与准确基线，GraphRAG 在一个连通的知识空间中沿着关系骨架展开导航式求解\cite{pan2024integrating}。随之而来的也有一些不足之处，在构建图时如果在上游抽取到偏差，它会沿着图的连接关系向下传导，使后续检索建立在失准的知识骨架之上，当课程知识图谱的规模随内容积累而扩张时，检索效率可能下降，课程资料的变化会导致图结构需要重新构建，更新成本高。

\section{提示词工程技术}
提示词工程是在不重新训练模型的前提下，通过设计输入文本的表达方式、上下文组织与输出约束，在推理阶段对模型行为实施外部控制。对于课程助教来说，知识边界、回答口径和输出格式都需要频繁调整，重新训练模型既不经济也难以跟上教学内容的更新节奏，提示词工程提供了一条不必改动模型权重即可将生成行为收束到课程框架内的可行路径。GPT-3 所展现的 few-shot 学习能力已初步表明，模型可以仅凭提示中的少量示例与任务说明完成迁移\cite{brown2020language}，InstructGPT 进一步证实经过指令对齐后的模型对自然语言任务约束的服从性更强，输出更为稳定\cite{ouyang2022training}。

提示词工程在方法层面整合了角色设定、任务定义、上下文补充、格式约束和示例演示等构件，对课程助教系统而言，角色设定将模型的回答身份锚定为“助教”，任务定义划定了回答的知识范围，格式约束则规范输出格式，抑制自由发挥带来的信息冗余或偏题。对于需要推理链条的问题，Chain-of-Thought 提示可以显式诱导模型输出中间推理步骤，从而提升复杂问题的求解能力\cite{wei2022chain}，Self-Consistency 则通过多条推理路径采样并综合结果，增强了推理输出的稳定性\cite{wang2022self}。提示词的作用远不止告诉模型做什么，还能进一步介入模型组织答案的方式和思考路径，它与检索模块紧密配合，将检索到的课程内容作为上下文嵌入提示词，使模型在生成阶段同时参照原始问题和外部知识，从而在机制上降低了幻觉风险\cite{yao2022react}。

\section{前后端开发技术}
前后端分离架构将用户界面、交互状态与内容呈现交由前端负责，为用户提供自然的交互体验，业务逻辑、数据组织、接口服务与模型调度则归入后端，承载知识检索、上下文组织和模型推理等计算密集任务，两者通过 HTTP 接口完成协作。这种分工方式降低了模块间的耦合度，便于各自独立维护、扩展和部署。

前端普遍采用组件化框架来组织界面，页面被拆解为相对独立的功能单元，各个模块可以分别开发和复用，在保证交互一致性的同时提升开发效率。随着系统复杂度上升，前端对类型约束和模块化构建的要求也在提高，TypeScript 在 JavaScript 语法之上引入静态类型描述，有助于约束接口字段、组件属性和状态对象的结构，减少大型交互系统中因类型混乱引入的前端逻辑错误。 Vite 等新一代构建工具则依托原生 ES Modules 提升开发阶段的启动速度与热更新效率，在生产环境下完成资源打包与优化。对需要持续迭代交互界面的课程助教系统来说，这套工具链能显著提高开发的灵活性。

课程助教系统的后端更看重接口清晰、并发处理和服务编排能力，轻量级 Web API 框架因此具有较高的适配性，以 FastAPI 为代表的现代 Python Web 框架利用类型标注来定义请求与响应结构，并结合自动校验机制提高接口开发效率和数据一致性，通过数据模型对输入输出进行约束能提升接口的可维护性，有助于前后端之间形成稳定的数据契约。

前后端之间的通信建立在基于 HTTP 的 RESTful 接口之上，前端通过异步请求将用户操作发往后端，后端完成业务处理后返回标准化的JSON数据。这种交互方式结构清晰、兼容性强，能够统一承载教材内容读取、会话创建、消息发送和历史重置等操作，并将这些操作模块化拆分，后续调整时不必整体重构界面层。

\section{数据库与知识库技术}

向量数据库在检索式问答系统中负责语义近邻检索，借助Sentence Transformers等嵌入模型把问题与文档映射到同一向量空间，通过相似度计算召回语义相近的片段，从而覆盖同一概念的不同问法或相近定义的改写。对课程助教系统来说，这一层能够有效承接学生提问中常见的口语化表达，构成知识召回的第一道滤网。课程的章节推进、概念间的先后依赖以及跨段落的论证关系，往往无法被孤立的文本片段所还原，Neo4j以节点、关系和属性组织数据，适合表达实体之间的关联结构，NetworkX则为图的构建、分析与算法处理提供了灵活的Python支持。在课程助教系统中，这类图结构被用来保存实体、关系与社区信息，使检索得以沿知识路径展开。

在教育场景中，数据库能够记录学生的提问行为和学习轨迹，为教学分析提供数据基础；知识库能够组织课程知识点，为智能助教系统提供可解释、可追溯的知识来源，两者协同运作，共同提升系统的可靠性和可用性，为大规模教学环境中的智能化支持提供关键技术保障。
